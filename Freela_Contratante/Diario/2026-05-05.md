## Sessao — fix/useRef-error: race condition na insercao otimista de vaga

**Bug investigado:** vaga criada em `criar-vaga.tsx` nao aparecia na home apos `router.back()`.

**Causa raiz identificada:** race condition no `useFocusEffect` da home.
- `addVaga(pending)` inseria a vaga otimisticamente no state local
- `fetchVagas()` era chamado logo em seguida (sem await)
- Quando o response da API chegava, `setVagas(data)` sobrescrevia o state inteiro, removendo a vaga inserida otimisticamente

**Correcao aplicada (`src/app/(home)/index.tsx`):**
Ordem invertida: `fetchVagas().then(() => { if (pending) addVaga(pending); })` — garante que a sincronizacao da API ocorra primeiro. Se a API ja retornou a vaga nova, a deduplicacao por `id` em `addVaga` a ignora. Se nao retornou (latencia de indexacao), ela e inserida manualmente.

**Validacao do contrato da API:**
- POST `/v1/{module}/vacancies` retorna `{ data: { ..., status: "OPEN", ... } }` — envelope desembrulhado pelo interceptor do axios
- Status `"OPEN"` e mapeado para `"aguardando"` pelo `vaga-status-map.ts` (via `.toLowerCase()`) — correto
- A vaga nova apareceria na secao "Vagas Abertas" da home — comportamento correto

**Arquivos alterados/criados:**

| Arquivo | Acao |
|---|---|
| `src/app/(home)/index.tsx` | Corrigido — `fetchVagas().then(() => addVaga(pending))` |
| `src/__tests__/utils/pending-vaga-store.test.ts` | Criado — 6 testes cobrindo comportamento do store |
| `src/__tests__/screens/home.test.tsx` | Atualizado — 3 novos testes validando ordem de execucao do useFocusEffect |

## Sessao — analise e correcao do fluxo completo de criacao de vagas

**Objetivo:** analise completa do fluxo de criacao de vagas contra o contrato da API.

### O que estava correto

| Ponto | Detalhe |
|---|---|
| Payload do POST | Campos `title`, `description`, `serviceType`, `date`, `startTime`, `endTime`, `address` corretos. `cityId` e opcional e o backend calcula pricing server-side |
| Interceptor do axios | Desembrulha `{ data: payload }` automaticamente — o response do POST ja chega desembrulhado ao service |
| Endpoint de listagem | `GET /v1/{module}/vacancies/contractors/:id` correto (primario) com `GET /v1/{module}/contractors/:id/vacancies` como fallback — ambos documentados no contrato |
| `mapApiStatus` com uppercase | `.toLowerCase()` antes do lookup ja tratava `"OPEN"` → `"aguardando"` corretamente |
| Insercao otimista | `fetchVagas().then(() => addVaga(pending))` com deduplicacao por `id` — correto |

### Problema corrigido — mapeamento de status incompleto

**Arquivo:** `src/utils/vaga-status-map.ts`

O `STATUS_MAP` nao cobria todos os status que a API pode retornar. Discrepancias identificadas:

| Status da API | Antes | Depois |
|---|---|---|
| `CLOSED` / `closed` | `"confirmado"` (errado semanticamente) | `"finalizado"` |
| `SCHEDULED` / `scheduled` | nao mapeado (caia em `"aguardando"`) | `"confirmado"` |
| `PAYMENT_PENDING` / `payment_pending` | nao mapeado (caia em `"aguardando"`) | `"aguardando"` (agora explicito) |
| `CANCELLED_BY_CONTRACTOR` | nao mapeado (caia em `"aguardando"`) | `"finalizado"` |
| `TRANSFER_PENDING` | nao mapeado (caia em `"aguardando"`) | `"finalizado"` |

**Nota semantica de `closed`:** no fluxo da API, `CLOSED` e o status que a vaga assume apos o check-out ser validado — ou seja, servico concluido. Estava mapeado como `"confirmado"` (secao "Proximas Contratacoes") quando deveria ir para `"finalizado"` (secao "Historico").
