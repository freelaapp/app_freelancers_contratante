## EAS Build & Submit â Fix build number duplicado (iOS e Android)

### Problema resolvido

O `eas submit` falhava com erro:
> *Build number 1 for app version 1.0.0 has already been used.*

### SoluĂ§ĂŁo

Adicionado `autoIncrement: true` no `eas.json` profile `production`:

```json
"production": {
  "autoIncrement": true,
  "android": { "autoIncrement": true },
  "ios": { "autoIncrement": true }
}
```

O EAS incrementa o `buildNumber` (iOS) e `versionCode` (Android) automaticamente em cada build.

### Comandos

```bash
# iOS
eas build --platform ios --profile production
eas submit --platform ios --profile production

# Android
eas build --platform android --profile production
eas submit --platform android --profile production

# Ambos juntos
eas build --platform all --profile production
```

## Sessao: feat/login-flow â fallbacks defensivos para shape da API

### Problema resolvido

O Swagger nao documenta o response body de `POST /v1/users/login`. O codigo original assumia `data.accessToken` e `data.user.*` sem fallback, causando crash se a API retornasse `token`, `access_token` ou o objeto user flat (sem `data.user`).

### Corrigo em `src/context/auth-context.tsx`

**`signIn`:**
- Token extraido com fallback em cadeia: `data.accessToken ?? data.access_token ?? data.token`
- Erro explĂ­cito lancado se nenhum dos tres estiver presente (facilita debug)
- User source com fallback: `data.user ?? data` (suporta tanto shape aninhado quanto flat)
- Todos os campos do usuario com `?? ""` / `?? false` para evitar `undefined` no estado

**`restoreSession`:**
- Mesmos `?? ""` / `?? false` nos campos de `UserMe` para consistĂȘncia defensiva

### Corrigido em `src/app/_layout.tsx`

**Guard de navegacao â bug de loop corrigido:**

Antes: `user && inAuthGroup` redirecionava para `completar-cadastro` mesmo quando o usuario ja estava nessa tela, potencialmente causando re-render em loop.

Depois: Logica separada em quatro casos mutuamente exclusivos:
1. Sem usuario + fora de `(auth)` â `/login`
2. Usuario logado + `!profileCompleted` + em `(auth)` + **nao** em `completar-cadastro` â `/completar-cadastro`
3. Usuario logado + `profileCompleted` + em `(auth)` â `/(home)`
4. Usuario logado + `!profileCompleted` + fora de `(auth)` â `/completar-cadastro`

### Corrigido em `src/types/api.ts`

**`LoginResponse`:** campos de token tornados opcionais (`accessToken?`, `access_token?`, `token?`, `refreshToken?`, `refresh_token?`) e `user?` opcional. Intersecao com `Partial<UserMe>` para suportar shape flat.

**`ConfirmEmailPayload`:** adicionado `userId?: string` e `email` tornado opcional, alinhando com o `ConfirmUserEmailDto` real do Swagger (apenas `code` e obrigatorio).
